# OMK 需求规格说明书

## 引言

OMK（`Kimi 编排与多智能体协调`，Oh-My-Kimi）是为 `Kimi Code CLI` 设计的多 `Agent`
编排框架。本文档为 OMK 四份中文技术文档中的需求规格说明书（REQ），定义 `0.1.0`
版本的功能需求、非功能需求、兼容性目标与功能边界，作为后续架构设计、系统设计与功能设计的基准与约束来源。

OMK 的核心目标是将多 `Agent` 协作、持久化状态管理与 `Skill` 编排体验引入 `Kimi Code
CLI`，同时保持零外部模型依赖，完全基于 Kimi 原生工具集运行。所有术语须与 `docs/zh/DOC_CONTRACT.md`
的术语表保持一致，并在正文中以反引号包裹。本文档的约束范围同时覆盖安装脚本与卸载脚本的行为需求，确保部署流程具备可预测性。

本文档约束的内容边界包括 `CLI 命令矩阵` 与 `错误码体系`，属于这两类关注点的规范源。其他关注点如 `Agent YAML
Schema`、`状态 Schema` 与 `配置项清单` 仅在本文档中以概括性说明引用，详细定义由后续文档承担。

## 文档范围与约束

本需求规格说明书仅描述 OMK 作为用户态扩展框架应具备的能力与约束，不涉及
`Kimi Code CLI` 本身的内部实现细节。所有功能需求均基于 Kimi CLI `1.36.0`
已公开的自定义接口进行推导，不包括对尚未发布特性的前瞻性假设。本文档的读者对象为 OMK
的核心贡献者、扩展开发者以及需要理解框架能力边界的技术负责人。

## 假设与依赖

OMK 的正常运行依赖 `Kimi Code CLI` `1.36.0` 或更高版本的已安装环境，且假设用户具备基本的命令行操作能力与
Python `>=3.9` 运行时环境。安装脚本假设目标系统支持 POSIX 标准 Shell，并提供 Bash 或 Zsh
作为默认解释器。通知功能假设用户能够自行获取对应平台所需的 Bot Token 或 Webhook 地址。

## 目标用户与用户画像

OMK 面向以下典型用户场景，每类场景对应不同的 `Skill` 与 `Agent` 组合需求：

1.
**全栈开发者**：需要快速原型构建或从零搭建服务，期望通过自然语言触发自主工作流完成端到端交付。典型调用包括
`autopilot`、`ralph` 等全自主执行模式，要求 `Agent`
具备完整的文件操作、命令执行与测试验证能力，并能在失败后自动重试直至任务完成。开发者期望以最小的人工干预获得可运行的代码仓库。

2. **代码审查工程师**：在合并前需要对复杂变更进行多维度质量门禁，包括逻辑、安全性与可维护性审查。依赖
`code-reviewer`、`security-reviewer` 等审查通道 `Agent` 与 `verify`
技能，要求审查结果结构化并可复现，同时支持在发现问题后自动指派修复任务给执行类
`Agent`。审查过程须保留审计日志以便追溯。

3. **技术负责人**：负责制定技术方案与项目规划，需要结构化访谈、风险评估与共识规划能力。依赖
`plan`、`ralplan`、`deep-interview`、`architect`
等规划与决策支持能力，要求输出可落地的技术方案、风险标记与实施顺序，并在团队内形成可共享的规划文档。规划文档须存放于标准化的规划目录中。

4. **DevOps 工程师**：需要并行执行构建、测试与部署任务，并在完成后通过外部通知渠道获得反馈。依赖
`ultrawork`、`team` 等并行编排技能，以及 `后台任务`
与通知集成，要求任务隔离、状态可查询，且失败任务能够独立重试而不影响其他并行管线。通知须覆盖
Telegram、Discord、Slack 三种渠道。

5. **开源贡献者**：在多个特性分支或 issue 之间切换，需要隔离的 `git worktree`
会话管理与持久化项目记忆。依赖 `project-session-manager` 技能与 `项目记忆`
机制，要求会话元数据跨分支持久化且可恢复，避免在切换上下文时丢失已完成的分析结论。贡献者期望通过自然语言指令快速创建或恢复开发会话。

6. **技术写作者**：需要维护大型文档体系，依赖知识库、记事本与跨 `会话` 项目记忆保持内容一致性。依赖
`wiki`、`writer-memory`、`remember` 等知识管理技能，要求 Markdown 知识库支持分层组织与检索，并能在多次 `会话`
之间继承已确认的术语表与风格指南。写作者希望将项目规范沉淀为可复用的结构化知识。

## 功能需求

### Skill 系统

OMK 提供 36 个 `Skill`，覆盖工作流编排、审查、诊断、知识管理与实用工具五大类别。`Skill`
以 `SKILL.md` 形式组织，通过自然语言触发词激活。`Skill` 须支持普通
`Skill` 与 `Flow Skill` 两种形态，后者用于定义多步骤自动化工作流。`Skill`
的安装与卸载须通过安装脚本和卸载脚本完成，支持全局、项目本地与指定目标目录三种部署粒度。安装器须验证目标目录是否为
Kimi 可识别的 `Skill` 搜索路径，并在发生冲突时提供覆盖或跳过选项。`Skill`
的元数据须包含名称、描述与兼容性说明，以便在 `Skill 发现` 阶段被正确注入 `系统提示词`。

### Agent 系统

OMK 定义 18 个公开专家 `Agent`，通过 `根 Agent` 统一调度。`子 Agent` 在独立上下文中运行，支持前台与
`后台任务` 两种执行模式。`Agent` 系统须支持 `Plan 模式` 与 `YOLO 模式`
两种审批策略，以适应不同风险偏好的场景。`Agent` 的调用契约与角色划分须与 `agents/default/agent.yaml`
中定义的公开名单保持一致，额外发布的配置文件除非接入已记录工作流，否则不属于支持接口。每个 `Agent`
应携带与其专业领域匹配的系统提示词约束，确保在编码、审查、规划或调试场景下的行为一致性。

### 状态管理

OMK 须在项目目录的 `状态目录` 中持久化运行时数据，包括模式状态、`项目记忆`、记事本、规划文档、审计日志与
`会话` 跟踪。状态管理支持跨 `会话` 恢复与查询，确保长周期任务的连续性。`状态目录`
的位置与结构须遵循 `README_ZH.md` 中定义的运行时路径约定。状态读写操作须通过 `omk state`
命令或等效模块接口完成，支持列出、读取、写入与清除四种基本操作。状态文件须采用人类可读的文本格式，便于调试与版本控制。

### CLI 工具

OMK 提供 `omk` 入口命令，覆盖状态读写、通知发送与更新检查等操作。安装脚本 `install.sh` 与卸载脚本 `uninstall.sh`
须支持全局安装、项目本地安装与指定目录安装三种模式，并提供预览、强制覆盖与跳过确认等交互选项。卸载脚本须支持保留运行时状态目录的选项，以防止误删用户数据。CLI
的退出码体系须在需求层面定义成功、参数错误、状态不存在等情形的行为，确保脚本化调用时的可观测性。安装器在执行前须显示将要安装或移除的文件清单。

### 通知集成

OMK 须支持通过环境变量配置 Telegram、Discord、Slack 三种外部通知渠道。通知功能由 `scripts/` 目录下的 Shell 脚本与
`omk notifier` 命令共同提供，用于报告 `后台任务` 完成状态、发送自定义消息推送，以及支持 `configure-notifications`
技能的引导式配置流程。通知消息须支持重试策略与错误降级，避免单次网络故障导致信息丢失。当多个通知渠道同时配置时，应支持选择性发送或广播发送两种策略，以满足不同场景的通知密度需求。

### 脚本与外部工具

`scripts/` 目录提供辅助 Shell
脚本，用于通知发送、更新检查、结构化代码搜索、语言服务器诊断与速率限制处理。`mcp-setup`
技能须支持引导式配置外部 CLI 工具，扩展 `Agent` 在浏览器自动化、代码检索等场景的能力边界。脚本执行须遵循
`Kimi Code CLI` 的 `Shell` 审批机制，不得绕过用户授权。所有脚本须具备跨平台兼容性，优先使用 POSIX
标准语法，并在非标准环境下给出明确错误提示。更新检查脚本应支持从远程仓库获取最新版本信息并与本地版本比对。

### 错误码体系

OMK CLI 须定义清晰的退出码规范：成功返回 `0`，参数错误或命令格式非法返回 `1`，状态文件不存在或模式未激活返回
`2`。Skill 执行失败时，须将错误状态回写至对应的状态文件，以便调用方通过 `omk state read`
查询失败原因。契约检查脚本在发现文档与实现不一致时，应返回非零退出码并输出具体的差异描述，便于持续集成流程捕获回归问题。退出码的设计须符合
Unix 惯例，便于 Shell 脚本进行条件分支处理。

## 非功能需求

### 准确性

`Skill` 触发词与 `Agent`
角色映射须保持精确，避免误触发或角色漂移。契约检查脚本须能验证文档、注册表、安装器消息与技能引用的一致性，确保所有事实清单项与客观数据保持同步。任何公开接口的变更须同步更新契约与引用，避免用户依赖已移除或未文档化的行为。触发词的设计应避免与常见自然语言表述过度重叠，降低误激活概率。

### 兼容性

OMK 仅依赖 `Kimi Code CLI` 原生能力，保持零 Claude 依赖。Python 运行时版本须满足
`>=3.9`，以确保在主流操作系统与长期支持环境中的可部署性。框架不得引入与 Kimi CLI 内置行为冲突的补丁或钩子。当
Kimi CLI 发布新版本时，OMK 应能在不修改核心代码的前提下，通过调整配置或 `Agent` 提示词完成适配。兼容性验证须以
`--agent-file` 启动路径为基准进行冒烟测试。

### 可维护性

文档与实现须遵循 `DOC_CONTRACT.md`
定义的术语表与引用规范，确保四份技术文档之间的术语一致性与内容边界清晰。`Skill` 与 `Agent`
的元数据变更须通过契约检查脚本进行回归验证，防止文档与实现脱节。代码仓库中的目录结构须与文档描述保持一一对应，降低新贡献者的理解成本。所有
`Skill` 的 `SKILL.md` 应保持合理篇幅，将详细内容移至辅助目录中以提升可读性。

### 安全性

`工具安全边界` 须遵循 `Kimi Code CLI` 的审批机制，`Shell` 命令执行、文件写入、`MCP` 工具调用与
`后台任务` 停止均需用户审批。工作区外文件操作须使用绝对路径并受额外约束，敏感文件（如 `.env`、SSH
私钥）始终被过滤，即使显式请求也不允许读取。通知相关的环境变量与令牌信息不得在日志或状态文件中以明文形式持久化。卸载脚本在移除文件时应明确提示用户，并提供保留状态目录的选项。

### 可扩展性

OMK 的技能目录与 `Agent` 配置须采用松耦合结构，允许用户在不修改核心框架的前提下，通过添加新的 `SKILL.md`
或继承现有 `Agent` 配置来扩展能力。扩展点须遵循 Kimi CLI 的加载优先级与继承规则，确保用户自定义内容与 OMK
内置内容可共存。当用户自定义的 `Skill` 与内置 `Skill` 同名时，加载行为须由 Kimi CLI 的目录优先级规则决定，OMK
本身不介入冲突仲裁。框架应为未来的通知渠道扩展预留接口，避免硬编码特定服务商逻辑。

## 功能边界

**OMK 负责：**

- 提供可安装、可卸载的 `Skill` 集合与 `Agent` 配置，并通过安装脚本管理部署生命周期；
- 管理项目级运行时状态与跨 `会话` 记忆，包括 `项目记忆`、记事本与知识库；
- 编排多 `Agent` 并行与串行工作流，支持前台执行与 `后台任务`；
- 提供通知与外部工具集成的引导能力，包括 Telegram、Discord、Slack 与环境变量配置；
- 维护公开 `Agent` 名单与 `Skill` 触发词的稳定接口契约，确保版本升级时的向后兼容；
- 提供契约检查与更新检查等运维辅助能力，保障框架自身的健康度与一致性。

**OMK 不负责：**

- 替代 `Kimi Code CLI` 本身或修改其内置行为、内置工具列表与审批机制；
- 提供独立的模型推理服务、`API 密钥` 管理或供应商路由；
- 实现图形化界面、IDE 插件或 `AI 聊天` 侧边栏集成；
- 托管远程 `MCP` 服务器、云端状态同步或多设备协作；
- 修改 Kimi CLI 的 `Skill 发现` 算法或目录优先级规则，仅适配并利用现有机制；
- 承担 Kimi CLI 自身的版本升级、错误修复或平台适配责任；
- 保证特定 `Agent` 输出内容的语义正确性，最终质量仍由底层模型能力与 `提示词` 工程共同决定。

## 兼容性需求

OMK 的兼容性目标为 `Kimi Code CLI` `1.36.0`。具体约束如下：

- `README_ZH.md` 的 `兼容性矩阵` 章节 定义了三种启动模式的支撑策略：纯 `kimi` 命令不在 OMK 支持范围内，因其不会加载
OMK 自定义 `根 Agent`；通过 `--agent-file` 加载自定义 `根 Agent` 是唯一的官方引导路径；仅内置 Kimi `子 Agent`
的启动模式属于部分兼容，可作为普通 Kimi 使用，但不构成 OMK 工作流接口。OMK 的验证流程须以 `--agent-file`
路径启动后调用自定义 `子 Agent` 作为最小可工作证明。

- `kimi-cli-docs/zh/customization/skills.md` 的 `Skill 发现` 章节 规定了 `内置 Skills`、`用户级
Skills` 与 `项目级 Skills` 的分层加载机制及目录优先级。OMK 的安装器须将技能部署到 Kimi
可识别的品牌组目录（如 `~/.kimi/skills/` 或 `./.kimi/skills/`），以确保正确的加载顺序。当存在同名 `Skill`
时，须遵循品牌组优先于通用组、项目级优先于用户级的解析规则。`merge_all_available_skills` 配置对 OMK
技能加载行为的影响须被视为外部配置，不在 OMK 控制范围内。安装器应避免向通用组目录写入，以防止与 Claude
或 Codex 生态的技能发生冲突。

- `kimi-cli-docs/zh/customization/agents.md` 的 `自定义 Agent 文件` 章节 说明了 `Agent` 配置文件的继承与覆盖机制。OMK
的 `根 Agent` 文件须通过 `--agent-file` 参数加载，且其定义的 `子 Agent` 路径须相对于该文件解析。`system_prompt_args`
等自定义参数须与 Kimi CLI 的内置变量协同工作，不得冲突。对继承机制的使用须确保 OMK 扩展的 `Agent` 能够正确复用
Kimi CLI 的默认工具集与安全策略，避免工具列表重复声明或遗漏关键安全约束。`子 Agent` 不可嵌套创建自己的 `子
Agent`，此限制由 Kimi CLI 原生施加，OMK 须在其文档与提示词中明确声明。
